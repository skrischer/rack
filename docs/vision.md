# Vision

## Problem

Today's fitness apps are closed silos. Coaching/AI logic, where it exists at
all, is baked in and not steerable; users cannot connect their own training
data to their own tools or agents. A lifter who wants a genuinely personal
trainer — one that reasons over their full history, designs plans their way,
and visualizes progress on demand — finds no app that exposes its data as a
programmable surface. The data is locked behind a fixed UI.

## Why now

MCP (Model Context Protocol) is established and MCP clients (Claude Desktop and
others) are in everyday use. The opening is to stop shipping yet another AI
feature inside an app and instead make the app an open, programmable
training-data platform that any MCP client can drive. Real-time socket sync
closes the loop between "the agent changed my plan" and "I see it in my app".

## Target users

A small, trusted circle (friends and family) of technically-minded lifters who
already use MCP clients and want their training plan and history to be
agent-steerable. Primary: the owner-lifter operating their own agent.
Secondary: the agent itself, as a first-class writer of the user's data.

## Goal

Rack is a native Android fitness app whose differentiator is not in-app AI but
an open, hosted rack-MCP. Each user has an account and a personal MCP API key;
any MCP client connects, designs training plans and visualization artifacts,
and writes them straight into the app through the MCP. Data syncs over a socket
so agent edits appear live and highlighted in the app. In-app, the focus is
consumption and logging — weight/reps/sets/RIR with history, in the Recomp
design language — while plan authoring is the agent's job.

## USP / differentiation

The one thing Rack does that the prior art does not: a hosted, per-user MCP
whose every write lands live and highlighted in a native app — programmable
training data with a visible real-time feedback loop, rather than a closed app
with a bolted-on read-only integration or a local single-user stdio MCP. The
differentiation evidence lives in `docs/prior-art.md` and its per-entry
ADOPT/AVOID harvest: reuse wger's CC-BY-SA exercise data but not its AGPL code;
invert the read-only-aggregation MCPs (Garmin/Whoop/Strava); surpass
wger-mcp's stdio/single-user/no-push ceiling; and deliberately skip CRDT
machinery the single-owner concurrency model does not need.

## Success criteria

- Round-trip latency: a write via rack-MCP appears in the app, highlighted, in
  < 1 s on the same network.
- Read-back consistency: an entry logged in the app is visible via the MCP in
  the client immediately.
- An MCP client can author a complete multi-day training plan and write it into
  a user's account in a single session.
- Native logging parity with the prototype: weight + per-set reps + RIR + dated
  history, persisted server-side.
- Per-user isolation: a user's API key can only read/write that user's data,
  verified by a cross-user access test.

## Scope

### In

- Native Android app: plan/day/exercise view, set logging (kg, reps, RIR),
  workout history.
- Hosted backend (VPS): accounts, per-user MCP API keys, training-data store
  seeded with the wger exercise catalog.
- rack-MCP: read + write tools over the user's plans, days, exercises, sets,
  and logs.
- Socket-based live sync with edit highlighting of agent writes.
- Visualization artifacts generated by the agent from training data.

### Out

- In-app plan authoring/editor — the agent authors.
- Nutrition, body metrics, social, wearable/health-platform integrations.
- iOS / web client.
- Formal data-protection / GDPR hardening.

## Non-goals

- Not a CRDT co-editing system — single owner plus their agent; a
  server-authoritative, versioned sync is enough.
- Not an AI-in-the-app product — intelligence lives in the user's own MCP
  client.
- Not a read-only fitness-data aggregator — Rack writes user-owned data, it
  does not slurp closed third-party feeds.
- Not a public multi-tenant SaaS — a friends-and-family circle on one VPS, no
  OAuth/compliance build-out.
